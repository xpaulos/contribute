# Building a Linux Container From Scratch: A Step-by-Step Guide

This tutorial guides you through creating, configuring, running, and observing a fully functional Linux container from scratch using nothing but native Linux kernel features.

No Docker, no Podman, and no container engines—just pure Bash and standard Linux utilities.

---

## Prerequisites & System Initialization

This guide assumes you are working inside **WSL2 (Windows Subsystem for Linux)** using **VS Code**, starting from a clean environment.

---

## Step 1: Install Debootstrap on Your Host

First, install `debootstrap`, a tool used to download and create a minimal Ubuntu filesystem.

### Fedora / Red Hat

```bash
sudo dnf install -y debootstrap
```

### Ubuntu / Debian

```bash
sudo apt update && sudo apt install -y debootstrap
```

---

## Step 2: Create the Workspace and Root Filesystem

Create a dedicated workspace for the project. This directory will contain both your project files and the container filesystem.

```bash
# Create and enter the project directory
mkdir -p ~/projects/diy-container
cd ~/projects/diy-container

# Option 1. Bootstrap a clean Ubuntu 24.04 (Noble Numbat) root filesystem
sudo debootstrap noble rootfs

# Option 2. Bootstrap a minimal Fedora 44 root filesystem
sudo dnf --installroot=$PWD/rootfs --releasever=44 --use-host-config --setopt=install_weak_deps=False install -y bash dnf fedora-release coreutils util-linux shadow-utils

```

> This process may take a few minutes because it downloads core system packages directly from Ubuntu mirrors.

# Entering the Container Sandbox

To isolate processes from the host, we'll use:

- `unshare` to create new namespaces
- `chroot` to change the apparent root filesystem

Run:

```bash
sudo unshare --mount --uts --ipc --pid --fork chroot rootfs /bin/bash
```

## Namespace Breakdown - 
🙍‍♂️*NOTE from human: Feel free to skip the deep diving, except for --pid and --mount (i guess)*


| Flag | Purpose |
|--------|---------|
| `--mount` | Isolates filesystem mounts |
| `--uts` | Isolates hostname/domain name |
| `--ipc` | Isolates IPC resources |
| `--pid` | Creates a new PID namespace |
| `--fork` | Starts a child process inside the namespace |

You are now running as `root` inside the isolated environment.

# Installing Python and Development Tools

Install Python, pip, virtual environment tooling, and Git:


> No `sudo` is required because you're already root inside the container.

---

## If you come across Package 'python3-pip' has no installation candidate 

### Troubleshooting: The "Universe" Repository Error

When installing `python3-pip`, you may encounter:

```text
E: Package 'python3-pip' has no installation candidate
```

### Why This Happens

By default, `debootstrap` only enables Ubuntu's **Main** repository.

Many Python-related packages are maintained in the **Universe** repository.

### Fix

Add Universe and update package indexes:

```bash
echo "deb http://archive.ubuntu.com/ubuntu noble main universe" > /etc/apt/sources.list

apt update
```

# Configure a Virtual Environment

Modern Linux distributions implement **PEP 668**, which discourages installing packages directly into the system Python environment.

Using a virtual environment is the recommended approach.

```bash
# Create the virtual environment
python3 -m venv venv_bare_min_container

# Activate it
source venv_bare_min_container/bin/activate
```

Install dependencies and start the application:

```bash
pip install -r <path_to_requirements.txt>

python3 <application_script_name>.py
```

> The terminal will remain occupied while the application is running.

# Observing the Container From the Host

Open a second terminal on the host machine to inspect the running container.

---

## 1. View Active Processes

Search for the running Python application:

```bash
ps aux | grep <application_script_name>.py
```

You should see the containerized Python process listed as a normal Linux process with a host PID (for example, `34414`) running under the root user.

---

## 2. Inspect Namespace Isolation

Linux provides the `lsns` utility for inspecting namespaces.

### PID Namespace

```bash
sudo lsns -t pid
```

You should observe your shell and Python process belonging to a PID namespace distinct from the host's primary namespace.

This prevents processes inside the container from seeing or interacting with host processes.

### Mount Namespace

```bash
sudo lsns -t mnt
```

You should find a mount namespace associated with the:

```bash
unshare ... chroot rootfs
```

command.

This namespace provides filesystem isolation.

### Network Namespace

```bash
sudo lsns -t net
```

Your container will not appear in a separate network namespace because `--net` was not used when creating namespaces using `unshare`.

As a result, the container shares the host's network stack.

This allows you to access your application directly from the host:

```text
http://localhost:<your_api_port>
```

# Core Container Architecture Concepts

## Namespace Lifecycle: Are Namespaces Ephemeral?

### Kernel Resources (Automatic Cleanup)

Linux namespaces are fully ephemeral.

The moment the last process inside the container exits (for example, after typing `exit` or terminating your Python application), the kernel automatically destroys the associated namespaces and reclaims all related resources.

No orphaned namespaces remain.

### Storage Resources (Manual Cleanup)

The `rootfs` directory is simply a normal folder stored on the host filesystem.

Packages and files installed inside it remain on disk until manually removed.

To delete the container filesystem:

```bash
rm -rf ~/projects/diy-container/rootfs
```

---

# The Missing Pillar: Cgroups

Namespaces define:

> What a container can see.

Cgroups define:

> What a container can use.

Examples include:

- CPU
- Memory
- Disk I/O
- Network bandwidth

In this tutorial, no cgroups are configured.

As a result, a runaway process inside the container (like `while True:`) could potentially consume all available CPU or memory on the host.

---

## Cgroups in Practice

Production container runtimes such as Docker combine namespaces with cgroups.

A simplified example:
🙍‍♂️*NOTE from human: didn't test it, just added it as placeholder to continue the exercise at some point*

```bash
# Create a memory control group
sudo cgcreate -g memory:my-custom-container

# Limit memory usage to 500 MB
sudo cgset -r memory.max=524288000 my-custom-container

# Start the container within that cgroup
sudo cgexec -g memory:my-custom-container \
    unshare --mount --uts --ipc --pid --fork \
    chroot rootfs /bin/bash
```

## Command Mapping: DIY Containers vs. Container Engines

To connect this exercise to industry-standard container tooling, the table below maps the raw Linux kernel commands you used manually to their automated Docker and Podman equivalents.

### Comparison Table

| Action / Feature                        | DIY Container (Manual Commands)                                                   | Docker Equivalent                               | Podman Equivalent                                  |
| --------------------------------------- | --------------------------------------------------------------------------------- | ----------------------------------------------- | -------------------------------------------------- |
| **Download & Build Root Filesystem**    | `sudo debootstrap noble rootfs`                          | `docker pull fedora:44` (or a Dockerfile build) | `podman pull fedora:44` (or a Containerfile build) |
| **Isolate Namespaces & Launch a Shell** | `unshare --mount --uts --ipc --pid --fork chroot rootfs /bin/bash`                | `docker run -it fedora:44 /bin/bash`            | `podman run -it fedora:44 /bin/bash`               |
| **Assign Container Hostname**           | `hostname bare-minimum-fedora`                                                    | `--hostname bare-minimum-fedora`                | `--hostname bare-minimum-fedora`                   |
| **Inject Application Files**            | `git clone <placeholder>` or manually copy files into `rootfs/`                   | `COPY . /opt/` (inside a Dockerfile)            | `COPY . /opt/` (inside a Containerfile)            |
| **Initialize Process Tracking**         | `mount -t proc proc /proc`                                                        | Handled automatically by the container runtime  | Handled automatically by the container runtime     |
| **Enforce Hardware Limits (Cgroups)**   | `cgcreate -g memory:my-container`<br>`cgset -r memory.max=524288000 my-container` | `-m 500m` or `--memory="500m"`                  | `-m 500m` or `--memory="500m"`                     |

## Key Takeaway

When you execute a command such as:

```bash
docker run -it \
    --network host \
    -m 500m \
    fedora:44 \
    /bin/bash
```

Docker is not performing any magic.

Under the hood, it translates that high-level command into the same Linux kernel primitives you used manually throughout this exercise:

* `unshare` → Namespace isolation
* `chroot` (or modern equivalents such as `pivot_root`) → Filesystem isolation
* `mount` → Virtual filesystem setup (`/proc`, `/sys`, etc.)
* `cgcreate` / `cgset` → Resource limits via cgroups

The container runtime then orchestrates these operations in the correct order and asks the Linux kernel to execute them.

# Summary

This tutorial demonstrated how to build a lightweight Linux container using only native kernel features:

1. Create a minimal filesystem using `debootstrap`.
2. Enter an isolated environment with `unshare` and `chroot`.
3. Configure the container environment.
4. Install and run an application.
5. Observe namespace isolation from the host.
6. Understand the distinction between namespaces and cgroups.
7. Side by side comparison with Container Engines

Modern container runtimes such as Docker automate these same building blocks:

1. **Namespaces** → Define what the container can see.
2. **Root Filesystems** → Define what files the container can access.
3. **Cgroups** → Define what resources the container can consume.
4. **Container Runtime** → Automates the setup and lifecycle management of the above.
